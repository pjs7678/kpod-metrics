eBPF-powered pod metrics
for Kubernetes

Lightweight DaemonSet that attaches eBPF programs to kernel tracepoints and exports per-pod CPU, network, memory, syscall, disk I/O, and filesystem metrics to Prometheus. Zero instrumentation. Zero sidecars.

Get Started View on GitHub

Why kpod-metrics?

Deep kernel observability without touching your application code.

Zero Instrumentation

eBPF attaches to kernel tracepoints automatically. No code changes, no sidecars, no agents inside your pods.

Per-Pod Granularity

Every metric is labeled with namespace, pod, container, and node. Drill down from cluster to individual container.

L7 Protocol Detection

Auto-detect HTTP, DNS, Redis, MySQL, Kafka, and MongoDB traffic with per-request latency and error tracking.

Prometheus-Native

Metrics exported directly to Prometheus. Works with Grafana dashboards (included), ServiceMonitor, and OTLP export.

Service Topology

Auto-discovered service dependency graph from TCP peer data. Shows latency, request rate, and protocol per edge. No configuration needed.

Kotlin eBPF DSL

BPF programs defined in type-safe Kotlin instead of C. Generates both kernel-side C code and Kotlin MapReader classes automatically.

Lightweight

~256 Mi memory per node. Tested up to 1,000 nodes / 100,000 pods. Three profiles (minimal, standard, comprehensive) to control overhead.

Architecture

Quick Start

Up and running in under a minute.

1

Add the Helm repository

helm repo add kpod-metrics https://pjs7678.github.io/kpod-metrics
helm repo update
2

Install

helm install kpod-metrics kpod-metrics/kpod-metrics \
--namespace kpod-metrics --create-namespace
3

Verify metrics

kubectl -n kpod-metrics port-forward ds/kpod-metrics 9090:9090
curl http://localhost:9090/actuator/prometheus | grep kpod

Try it locally with kind

One command to spin up a local demo cluster:

git clone https://github.com/pjs7678/kpod-metrics.git
cd kpod-metrics && ./scripts/quickstart.sh

Built Different: Kotlin eBPF DSL

Define eBPF programs in type-safe Kotlin instead of writing raw C. The DSL generates both kernel-side C code and Kotlin MapReader classes.

Kotlin DSL (what you write)
val memProgram = ebpfProgram("mem") {
  val counterKey = struct("counter_key") {
    u64("cgroup_id")
  }
  val oomKills = hashMap(
    "oom_kills", counterKey,
    BpfScalar.U64, maxEntries = 10240
  )

  tracepoint("oom", "mark_victim") {
    val cgId = getCurrentCgroupId()
    val ptr = mapLookupElem(oomKills, cgId)
    ifNonNull(ptr) { atomicIncrement(it) }
  }
}
Generated (what runs in the kernel)
struct counter_key {
  __u64 cgroup_id;
};

struct {
  __uint(type, BPF_MAP_TYPE_HASH);
  __uint(max_entries, 10240);
  __type(key, struct counter_key);
  __type(value, __u64);
} oom_kills SEC(".maps");

SEC("tp/oom/mark_victim")
int handle_mark_victim(void *ctx) {
  struct counter_key key = {};
  key.cgroup_id = bpf_get_current_cgroup_id();
  __u64 *val = bpf_map_lookup_elem(&oom_kills, &key);
  if (val) __sync_fetch_and_add(val, 1);
  return 0;
}
